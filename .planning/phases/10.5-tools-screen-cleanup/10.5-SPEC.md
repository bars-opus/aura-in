# Phase 10.5 — Tools Screen Cleanup Sprint

## Outcome

The Tools tab of the shop-owner dashboard is the most-used surface in the
daily workflow and is currently broken in production. After this phase:

- Every card in [tools_screen.dart](../../../lib/presentation/features/shops/dashboard/presentation/screens/tools_screen.dart)
  routes to the correct destination.
- The "Send manual reminder" code path stops calling a non-existent edge
  function (`send-reminder`) and instead routes through the existing
  `whatsapp-send` + `scheduled_notifications` infrastructure used by the
  booking system.
- Three ghost tables and one ghost RPC (`promotions`, `promotion_redemptions`,
  `shop_settings`, `increment_promotion_usage`) gain backfill migrations so
  source control matches production.
- Error leakage in `RemindersController`, `PromotionsController`, and
  `ExportController` is sanitized to match the pattern we just shipped for
  the other dashboard controllers (stable error code + `AppLogger.warn`).
- The N+1 in promotion stats is collapsed into one SQL query.
- `incrementUsage` becomes atomic via a SECURITY DEFINER RPC, closing the
  compensating-tx gap.

**This is a stop-the-bleeding sprint, not a feature phase.** No new tools,
no new UI surfaces beyond fixing what's broken. Feature work (multi-stage
reminders, retention engine, promo engine upgrade) lives in Phases 11–14.

## Why this matters

The Tools tab is what owners poke during the actual workday. Three of six
cards land on the wrong screen, one (Reminder Manual Send) silently fails
on every tap. Phase 10 made the Analytics tab decision-grade; Phase 10.5
brings the Tools tab to "doesn't actively mislead users."

- **No new tools.** No Cancellation Policy editor, no Birthday auto-send, no marketing broadcast, no loyalty/gift cards. Those are Phases 11+.
- **No upgrades to the Reminder engine itself** — single-stage today, single-stage after this phase. Multi-stage is Phase 12.

- **Money-as-minor-units.** Promotion `discount_amount` stays `double` for parity with the rest of the dashboard.

## Definitions

- **Dead route**: a card whose `onTap` opens a screen unrelated to the card's label. Currently: Payment Settings → Export, Business Hours → Export, Service Management → Export.
- **Ghost object**: a table, view, or function called by Dart code but with no `CREATE` statement in `supabase/migrations/`. Same risk surface as the `performance_alerts` / `shop_wallets` cluster we just closed.
- **Stable error code**: a short snake_case string the UI maps to fixed copy. Never `e.toString()`. Examples: `'load_failed'`, `'invalid_input'`, `'duplicate_code'`.

## Reminder send path rewiring

Current path ([supabase_dashboard_repository.dart:2347-2360](../../../lib/presentation/features/shops/dashboard/data/repositories/supabase_dashboard_repository.dart#L2347)):

```dart
await _supabase.functions.invoke(
  'send-reminder',
  body: {'booking_id': bookingId, 'type': 'manual'},
);
```

The `send-reminder` function does not exist in `supabase/functions/`. Every call silently 404s, the controller catches the exception, and the UI shows nothing (or, after this phase's other fix, an `AppLogger.warn`).

New path: insert directly into `scheduled_notifications` with `scheduled_for = now()`. The existing `process-scheduled-notifications` cron picks it up within ~60 seconds and dispatches via the configured channels.

This must happen via a new SECURITY DEFINER RPC (not a direct INSERT from the client) because the client doesn't have permission to write `scheduled_notifications` for arbitrary `(user_id, booking_id, shop_id)` triples, and we need to enforce that the caller owns the shop the booking belongs to.

```sql
CREATE OR REPLACE FUNCTION public.schedule_manual_booking_reminder(
  p_booking_id UUID
) RETURNS UUID
LANGUAGE plpgsql SECURITY DEFINER SET search_path = public
AS $$
DECLARE
  v_booking RECORD;
  v_owns BOOLEAN;
  v_notification_id UUID;
BEGIN
  SELECT b.id, b.user_id, b.shop_id, b.start_time, s.user_id AS shop_owner_id
    INTO v_booking
  FROM bookings b
  JOIN shops s ON s.id = b.shop_id
  WHERE b.id = p_booking_id;

  IF NOT FOUND THEN
    RAISE EXCEPTION 'not_found' USING ERRCODE = 'P0002';
  END IF;

  IF v_booking.shop_owner_id IS DISTINCT FROM auth.uid() THEN
    RAISE EXCEPTION 'not_found' USING ERRCODE = '42501';
  END IF;

  IF v_booking.user_id IS NULL THEN
    -- Guest booking: cannot schedule a push reminder. WhatsApp path
    -- via whatsapp-send would need a phone — that's the bulk path.
    RAISE EXCEPTION 'guest_booking_not_supported'
      USING ERRCODE = 'P0001', HINT = 'NO_PUSH_FOR_GUEST';
  END IF;

  INSERT INTO scheduled_notifications (
    user_id, notification_type, booking_id, shop_id,
    scheduled_for, metadata
  ) VALUES (
    v_booking.user_id,
    'booking_reminder_manual',
    v_booking.booking_id,
    v_booking.shop_id,
    now(),
    jsonb_build_object(
      'title',  'Upcoming appointment',
      'body',   'Your appointment is coming up.',
      'source', 'manual_owner_send'
    )
  )
  RETURNING id INTO v_notification_id;

  RETURN v_notification_id;
END;
$$;

REVOKE ALL ON FUNCTION public.schedule_manual_booking_reminder(UUID) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.schedule_manual_booking_reminder(UUID) TO authenticated;
```

The repository method `sendManualReminder(String bookingId)` becomes a one-line wrapper that calls the RPC. `sendBulkReminders` becomes a server-side `INSERT INTO scheduled_notifications SELECT ...` RPC that runs once per shop, replacing the loop-of-N-RPC-calls pattern that exists today.

Caveat: this only delivers push notifications today. WhatsApp delivery is a configuration of `process-scheduled-notifications` — out of scope for this phase. If the user's notification preferences don't allow push, the worker `skipped`s the row, which is the existing behavior.

## Backfill migrations

Same byte-for-byte pattern as Phase 0's `20260603000400_backfill_performance_alerts.sql`:

1. **`promotions`** — columns lifted from prod. Includes the `code` UNIQUE constraint per shop (`UNIQUE (shop_id, code)`).
2. **`promotion_redemptions`** — FK to `promotions(id)` + `bookings(id)`, RLS allowing shop owners to read their own.
3. **`shop_settings`** — JSONB blob storage used for `reminder_settings` and presumably future config. Single row per shop, RLS scoped to owner.
4. **`increment_promotion_usage(p_promotion_id UUID)`** — current SECURITY DEFINER signature; behavior to be replaced by the consolidated RPC below.

These four backfills are no-ops against the live DB (everything uses `CREATE TABLE IF NOT EXISTS` / `CREATE OR REPLACE FUNCTION`) — they exist to make CI / local / staging match prod.

## Atomic promotion redemption RPC

Replace the current two-step pattern in [promotions_repository.dart:86-97](../../../lib/presentation/features/shops/dashboard/data/repositories/promotions_repository.dart#L86) with a single SECURITY DEFINER RPC that does the counter increment + redemption insert + idempotency check inside one transaction. This closes the compensating-tx gap (checklist 1.10): today, if the RPC succeeds and the INSERT fails, the promotion counter is over-counted relative to the redemption ledger.

```sql
CREATE OR REPLACE FUNCTION public.redeem_promotion(
  p_promotion_id    UUID,
  p_booking_id      UUID,
  p_user_id         UUID,
  p_discount_amount NUMERIC
) RETURNS UUID
LANGUAGE plpgsql SECURITY DEFINER SET search_path = public
AS $$
DECLARE
  v_existing_id UUID;
  v_promo_row RECORD;
  v_redemption_id UUID;
BEGIN
  -- Idempotency: a single (promotion, booking) pair is at most one redemption.
  SELECT id INTO v_existing_id
  FROM promotion_redemptions
  WHERE promotion_id = p_promotion_id AND booking_id = p_booking_id;
  IF v_existing_id IS NOT NULL THEN
    RETURN v_existing_id;
  END IF;

  SELECT * INTO v_promo_row FROM promotions WHERE id = p_promotion_id FOR UPDATE;
  IF NOT FOUND THEN
    RAISE EXCEPTION 'not_found' USING ERRCODE = 'P0002';
  END IF;

  IF v_promo_row.usage_limit IS NOT NULL
     AND v_promo_row.usage_count >= v_promo_row.usage_limit THEN
    RAISE EXCEPTION 'limit_reached'
      USING ERRCODE = 'P0001', HINT = 'PROMO_LIMIT_REACHED';
  END IF;

  IF p_discount_amount IS NULL OR p_discount_amount <= 0 THEN
    RAISE EXCEPTION 'invalid_amount' USING ERRCODE = '22023';
  END IF;

  UPDATE promotions
     SET usage_count = usage_count + 1, updated_at = now()
   WHERE id = p_promotion_id;

  INSERT INTO promotion_redemptions (
    promotion_id, booking_id, user_id, discount_amount
  ) VALUES (
    p_promotion_id, p_booking_id, p_user_id, p_discount_amount
  )
  RETURNING id INTO v_redemption_id;

  RETURN v_redemption_id;
END;
$$;

REVOKE ALL ON FUNCTION public.redeem_promotion(UUID, UUID, UUID, NUMERIC) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.redeem_promotion(UUID, UUID, UUID, NUMERIC) TO authenticated;
```

The existing `increment_promotion_usage` is left in place but marked deprecated in its `COMMENT ON FUNCTION` text — anyone who still calls it gets a noop counter bump, but the new code path uses `redeem_promotion` exclusively.

## N+1 collapse for `_getTotalRevenueImpact`

Current:

```dart
for (final promotion in promotions) {
  final response = await _supabase
      .from(_redemptionsTable)
      .select('discount_amount')
      .eq('promotion_id', promotion.id);
  // sum...
}
```

Replacement (single Postgrest call):

```dart
final response = await _supabase
    .from(_redemptionsTable)
    .select('promotion_id, discount_amount')
    .inFilter('promotion_id', promotionIds);
// group + sum client-side
```

The shop typically has < 50 promotions; client-side aggregation is fine here. A future phase can move this into the `get_promotion_stats` RPC if it ever becomes a problem.

## Required fixes before execution

1. **Task 1.5 ‚Äî `redeem_promotion` is missing null-guards on `p_promotion_id` and `p_booking_id`.** Problem: a NULL promotion id falls through to the `SELECT EXISTS (SELECT 1 FROM public.promotions p JOIN public.shops s ...)` ownership check, which returns FALSE for any NULL, and raises `not_found / 42501`. This masks a client-side bug as a security error and violates checklist 2.1 (input sanitization: format/type). Change: add `IF p_promotion_id IS NULL OR p_booking_id IS NULL THEN RAISE EXCEPTION 'invalid_input' USING ERRCODE = '22023', HINT = 'IDS_REQUIRED'; END IF;` immediately after `BEGIN` and before the `p_discount_amount` validation (`10.5-PLAN.md:351‚Äì356`). Update Task 6.1 ¬ßg to also test null inputs.

2. **Task 1.4 ‚Äî `schedule_bulk_manual_booking_reminders` references `b.booking_date`, an unverified column.** Problem: `10.5-PLAN.md:316` filters `WHERE b.booking_date = p_date`. RESEARCH never confirmed `bookings` has a `booking_date` column; the heatmap RPC at `20260603001500_harden_dashboard_rpcs.sql:66` uses `start_time::DATE`. If the column is named differently in prod (or doesn't exist), the bulk RPC fails on first call. Change: replace `AND b.booking_date = p_date` with `AND b.start_time::DATE = p_date` to match the heatmap pattern, OR add a Task 1.0a step that runs `SELECT column_name FROM information_schema.columns WHERE table_name='bookings' AND column_name='booking_date';` against staging and locks the result before writing Task 1.4. The smoke-test (Task 6.1 ¬ßd) will catch this only after the migration is already pushed.

3. **Task 1.5 ‚Äî `redeem_promotion` validation runs before authz, deviating from the hardening template.** Problem: `10.5-PLAN.md:353‚Äì367` validates `p_discount_amount` before the `SELECT EXISTS` ownership check. The template at `20260603001500_harden_dashboard_rpcs.sql:44‚Äì59` runs ownership FIRST, then validation. The deviation lets a non-owner caller probe with `p_discount_amount = 1` to distinguish "you don't own this promo" (not_found/42501) from "you do own it but the limit is reached" (limit_reached/P0001). Change: move the `p_discount_amount` null/positive check to AFTER the ownership `SELECT EXISTS` block but BEFORE the `FOR UPDATE` lock. This restores ordering parity with the rest of the codebase.

4. **Task 4.1 ‚Äî verification grep is too permissive.** Problem: `grep -n 'e.toString()' ... | grep -v "fields:" | wc -l` (`10.5-PLAN.md:495`) filters out any line containing the literal `fields:`, which can falsely clear a line that does both log AND leak in the same Dart statement. Change: replace with `grep -n "error: e.toString()" lib/presentation/features/shops/dashboard/presentation/controllers/reminders_controller.dart | wc -l` and assert `0`. Same fix for Tasks 4.2 and 4.3 grep gates.

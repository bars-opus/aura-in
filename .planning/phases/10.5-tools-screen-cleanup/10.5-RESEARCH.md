# Phase 10.5 Research тАФ Tools Screen Cleanup

**Researched:** 2026-06-04
**Domain:** Flutter dashboard cleanup + Supabase RPC / migration backfill
**Confidence:** HIGH (everything cross-referenced against live files)

## Summary

The spec is **largely correct** but assumes several things that the source disagrees with. The four corrections the planner must apply before writing PLAN.md:

1. The DTO field name on `shopDetailsProvider` is `userId` and `currency`, not `shopOwnerId` and `shopCurrencyCode` тАФ the planner must spell out the mapping so the engineer doesn't go hunting.
2. `scheduled_notifications` already has a `delivery_channel` column (push / whatsapp / none) added in [20260528120000_link_booking_guest_support.sql:89-95](../../../supabase/migrations/20260528120000_link_booking_guest_support.sql#L89). The new RPC must set it explicitly or rely on the default `'push'`. The spec's RPC omits it тАФ needs to be added.
3. The worker's `notification_type` is *probably* an enum in prod (per [20260602130000_add_notification_type_enum_values.sql:6-9](../../../supabase/migrations/20260602130000_add_notification_type_enum_values.sql#L6)). Inserting `'booking_reminder_manual'` will raise `invalid input value for enum notification_type` unless we add it via `ALTER TYPE ... ADD VALUE IF NOT EXISTS`. The spec misses this.
4. The worker calls `notification_settings.push_enabled` only (single global gate), not `booking_reminders_enabled`. The spec's "consent gate" question has a one-line answer: the existing worker already handles it, and a per-user opt-out auto-becomes `status='skipped'`. We do NOT need to add consent logic to the new RPC.

## Findings

### 1. PaymentSettingsScreen invocation shape

The route is real and definitely takes a `Map<String, String>` extra with five keys ([app_router.dart:774-787](../../../lib/app/routing/app_router.dart#L774)). All five values *are* available from `ToolsScreen`'s existing `shopDetailsProvider(shopId)` watch тАФ but **the field names differ from the route's `extra` keys**:

| Route extra key | DTO field on `ShopDetailsDTO` | Notes |
|-----------------|-------------------------------|-------|
| `shopId` | (passed in to `ToolsScreen`) | from widget constructor |
| `shopName` | `shop.shopName` | [shop_details_dto.dart:13](../../../lib/presentation/features/shops/query/data/models/dtos/shop_details_dto.dart#L13) |
| `shopOwnerId` | `shop.userId` | [shop_details_dto.dart:21](../../../lib/presentation/features/shops/query/data/models/dtos/shop_details_dto.dart#L21) тАФ DTO calls it `userId`, route calls it `shopOwnerId` |
| `shopCurrencyCode` | `shop.currency` | [shop_details_dto.dart:29](../../../lib/presentation/features/shops/query/data/models/dtos/shop_details_dto.dart#L29) тАФ nullable, JSON falls back through `'currency' ?? 'currency_code'` (line 211-212). Default to empty string per the route's null-coalescing pattern. |
| `shopCountry` | `shop.country` | [shop_details_dto.dart:28](../../../lib/presentation/features/shops/query/data/models/dtos/shop_details_dto.dart#L28) тАФ nullable |

The Payment Settings card tap must therefore be inside `shopDetailsAsync.maybeWhen(data: (shop) => ...)` тАФ tapping while `shopDetailsAsync.isLoading` or `.hasError` must either be a no-op or show a "still loading" SnackBar. Currently `ToolsScreen` only reads `shopName` and `bookingSlug` from the async тАФ it never branches on loading/error for the cards themselves. Planner should make the Payment Settings card disabled-while-loading.

No new providers needed. `shopDetailsProvider(shopId)` is sufficient.

Confirmed against [20260507000000_notification_engine.sql:18-32](../../../supabase/migrations/20260507000000_notification_engine.sql#L18) plus the link-booking extensions in [20260528120000_link_booking_guest_support.sql:89-95](../../../supabase/migrations/20260528120000_link_booking_guest_support.sql#L89):

| `whatsapp_params` | jsonb | yes | тАФ |

**Metadata shape the worker expects:** the worker at [process-scheduled-notifications/index.ts:271-275](../../../supabase/functions/process-scheduled-notifications/index.ts#L271) destructures `{ title, body, ...rest }` from `metadata` and throws if either is missing. The spec's RPC sets `title='Upcoming appointment'`, `body='Your appointment is coming up.'` тАФ meets the contract. **But** if we want the push to navigate the user to the booking detail when tapped, we should include `booking_id` in `rest` so `data` carries it; OneSignal `data` payload is what RN/Flutter background handlers read.

**`'booking_reminder_manual'` enum value:** in prod the column is an enum (per the diagnostic in 20260602130000). We need a migration that runs `ALTER TYPE notification_type ADD VALUE IF NOT EXISTS 'booking_reminder_manual'` *with the same DO $$ guard pattern* the existing migration uses so it's safe against the TEXT case. Without it, the new RPC's INSERT fails on prod with `invalid input value for enum notification_type`.

### 3. process-scheduled-notifications worker contract

Walked the whole file. Key facts:

- **Two delivery branches:** [index.ts:255](../../../supabase/functions/process-scheduled-notifications/index.ts#L255) checks `delivery_channel === 'whatsapp'` тЖТ WhatsApp path. Otherwise тЖТ OneSignal push. Default is `'push'`. Manual reminders fired by shop owners should leave `delivery_channel` unset (defaults to push) тАФ WhatsApp requires phone + template which we don't have for arbitrary shop-side sends.
- **Push consent gate:** [index.ts:262-269](../../../supabase/functions/process-scheduled-notifications/index.ts#L262) calls `isPushEnabled(user_id)` which reads `notification_settings.push_enabled` only. If false, the row is marked `'skipped'` and never delivered. **The worker already handles consent end-to-end тАФ the new RPC does not need a `notification_preferences` check.** The right answer to spec's Question 7 is "noop, the worker covers it."
- **No fallback pushтЖТWhatsApp:** if push fails, the worker exponentially backs off with retry_count++. It never falls back to WhatsApp. WhatsApp delivery only triggers when the row was inserted with `delivery_channel='whatsapp'` (and a `whatsapp_template` + `whatsapp_params`).
- **No-row case for `notification_settings`:** [index.ts:55-64](../../../supabase/functions/process-scheduled-notifications/index.ts#L55) returns `data?.push_enabled !== false`, i.e., absence of a settings row defaults to opt-in. There's also an `on_auth_user_created_notification_settings` trigger ([20260507000000_notification_engine.sql:117-119](../../../supabase/migrations/20260507000000_notification_engine.sql#L117)) so every user has a row anyway.
- **Status values written:** `pending`, `processing`, `sent`, `failed`, `skipped`, `cancelled` тАФ matches spec.

### 4. shop_settings table reality

Grep confirms only two callsites both in [supabase_dashboard_repository.dart:2284-2344](../../../lib/presentation/features/shops/dashboard/data/repositories/supabase_dashboard_repository.dart#L2284):

**Only columns referenced:** `shop_id`, `reminder_settings` (JSONB), `updated_at`. The backfill is therefore trivial:

```sql
ALTER TABLE public.shop_settings ENABLE ROW LEVEL SECURITY;
-- RLS: shop owner can read/write their own row (auth.uid() = shops.user_id)
```

The `20260603001100_rewrite_check_daily_withdrawal_limit.sql:35` TODO comment about sourcing `v_daily_limit` from `shop_settings` is **not** an actual reference тАФ it's a future-work note. No other columns needed.

### 5. promotions + promotion_redemptions columns

`Promotion.toJson()` ([promotion_model.dart:110-124](../../../lib/presentation/features/shops/dashboard/data/models/promotion_model.dart#L110)) defines the prod contract:

```sql
CREATE TABLE IF NOT EXISTS public.promotions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  shop_id         UUID NOT NULL REFERENCES public.shops(id) ON DELETE CASCADE,
  name            TEXT NOT NULL,
  code            TEXT NOT NULL,
  discount_type   TEXT NOT NULL CHECK (discount_type IN ('percentage','fixed','free_addon')),
  discount_value  NUMERIC NOT NULL,
  valid_from      DATE NOT NULL,         -- model uses .toIso8601String().split('T').first тЖТ DATE not TIMESTAMPTZ
  valid_to        DATE NOT NULL,         -- same тАФ spec says timestamptz, model says date
  usage_limit     INT,                   -- nullable (model: `int?`)
  usage_count     INT NOT NULL DEFAULT 0,
  is_active       BOOLEAN NOT NULL DEFAULT true,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (shop_id, code)
);
```

**Correction to spec:** `valid_from` / `valid_to` are DATE, not TIMESTAMPTZ тАФ the model strips time with `.split('T').first` on the write side and parses with `DateTime.parse` on the read side. If the planner backfills as TIMESTAMPTZ, the existing prod rows still work (PG coerces) but new writes from the Dart side will send dates. Either backfill as DATE (matches model intent) or leave as TIMESTAMPTZ (matches what's likely in prod) тАФ verify against `SELECT data_type FROM information_schema.columns WHERE table_name = 'promotions'` on staging before locking. Mark this as a **P0 planner question**.

**No created_by column.** RLS is owner-scoped via `shops.user_id = auth.uid()` join, same pattern as performance_alerts.

`promotion_redemptions` per [promotions_repository.dart:92-97](../../../lib/presentation/features/shops/dashboard/data/repositories/promotions_repository.dart#L92): columns written are `promotion_id`, `booking_id`, `user_id`, `discount_amount`. Spec is correct. Add `id UUID PK`, `created_at TIMESTAMPTZ DEFAULT now()`, and the **uniqueness constraint** that makes `redeem_promotion`'s idempotency work: `UNIQUE (promotion_id, booking_id)`. The spec's idempotency check does a `SELECT FOR UPDATE`-less lookup; relying on the UNIQUE constraint + `ON CONFLICT DO NOTHING ... RETURNING id` would be cleaner, but the spec's pattern is fine if we keep the explicit lock on the parent `promotions` row.

### 7. notification_preferences interaction (consent)

The relevant table is `notification_settings`, not `notification_preferences` (the spec's wording is slightly off). The worker reads only `push_enabled` ([process-scheduled-notifications/index.ts:55-64](../../../supabase/functions/process-scheduled-notifications/index.ts#L55)) тАФ not `booking_reminders_enabled`. The other booleans (`booking_reminders_enabled`, `marketing_enabled`, `new_shops_nearby_enabled`) are declared in [20260507000000_notification_engine.sql:89-97](../../../supabase/migrations/20260507000000_notification_engine.sql#L89) but **nothing in the worker reads them**.

**Implication for the spec's RPC:** the worker already covers global push opt-out via `status='skipped'`. The RPC does NOT need to gate on `booking_reminders_enabled`. If the planner wants per-category consent (which would be the more correct PII/consent posture), that's a **worker-side change** (one extra column read in `isPushEnabled`), not an RPC-side change тАФ and that change is out of scope for this stop-the-bleeding sprint. **Recommend: explicitly note in the PR description that manual reminders respect global push opt-out only, and file a P2 ticket to gate on `booking_reminders_enabled` in the worker.**

### 8. Coming Soon disabled-card UX precedent

`grep -rn "'Coming Soon'" lib/` returns zero results outside `tools_screen.dart`. **No precedent exists in the codebase.** The simplest mechanic that matches the design tokens:

- Reduce `KpiCard` opacity to ~0.5 (wrap in `Opacity` or pass an `enabled: false` parameter).
- Set `onTap` to a closure that shows a SnackBar via the existing `SnackBarUtils` / `SnackbarWidget` (path: search `lib/core/widgets/feedback/snackbar_widget.dart` тАФ already touched in this branch per git status).
- Keep the "Coming Soon" title verbatim.

Snackbar copy: `'Coming in a future release.'` Use the project's existing `error: false` / `warning: true` variant so it doesn't look like a failure.

**Alternative considered:** wrap in `IgnorePointer` + `Opacity`. Rejected because then the SnackBar can't fire тАФ and showing the user "this is coming soon" on tap is the actual UX intent. Wrap the existing `onTap` instead.

KpiCard currently has no `enabled` param тАФ adding it is one extra prop on the widget (~3 lines). Planner should treat this as a small `KpiCard` API change, not a separate refactor.

### 9. Other risks the spec missed

| # | Risk | Severity | Mitigation |
|---|------|----------|------------|
| R1 | **Enum-vs-text mismatch on `notification_type`.** Spec inserts `'booking_reminder_manual'` without `ALTER TYPE`. Will 500 on every call in prod. | P0 тАФ blocks the whole reminder path | Add `20260604000050_add_booking_reminder_manual_enum.sql` mirroring the 20260602130000 DO $$ guard pattern. Land it before the RPC migration. |
| R2 | **Past-time bookings.** Spec doesn't guard against scheduling a manual reminder for a booking that already started or is in the past. Sending "Your appointment is coming up" to a customer mid-haircut is bad. | P1 тАФ UX | Add `IF v_booking.start_time <= now() THEN RAISE EXCEPTION 'booking_in_past' USING ERRCODE = 'P0001'; END IF;` to `schedule_manual_booking_reminder`. |
| R3 | **`sendBulkReminders` rate-limit risk.** No daily cap. If a shop has 200 bookings and the owner taps "remind all" three times, that's 600 pushes. OneSignal won't refuse but it'll cost. | P1 тАФ money/UX | Add a server-side dedup: `INSERT INTO scheduled_notifications ... ON CONFLICT (booking_id, notification_type) WHERE notification_type='booking_reminder_manual' AND status IN ('pending','processing') DO NOTHING`. Requires a partial unique index. Document as opt-in if too complex; otherwise the owner sees no error and silently bills nothing extra. |
| R4 | **Spec's `redeem_promotion` lock order vs. idempotency check.** Idempotency `SELECT` runs *before* `SELECT ... FOR UPDATE` on `promotions`. Under heavy concurrency two threads could both read NULL existing redemption then both pass the limit check. The UNIQUE constraint on `(promotion_id, booking_id)` would catch the second INSERT but only after the counter was already bumped twice. | P1 тАФ correctness | Move the existing-redemption check to AFTER the `FOR UPDATE` on `promotions`, OR rely on the UNIQUE constraint and do `INSERT ... ON CONFLICT DO NOTHING RETURNING id`, then conditionally bump usage_count only if the insert produced a row. Recommend the latter тАФ single statement, no race. |
| R5 | **Guest booking on `redeem_promotion`.** Spec requires `p_user_id` non-null but bookings can be guest (post-20260528120000). Need to either accept `NULL` `user_id` or fail fast with a stable code. | P2 | Add `p_user_id UUID` тЖТ make column nullable in `promotion_redemptions`, or `RAISE EXCEPTION 'guest_redemption_unsupported'` if NULL. |
| R6 | **`schedule_manual_booking_reminder` authz uses `bookings тЖТ shops.user_id`.** The codebase has multi-owner workers via `shop_workers` table (Phase 1 docs). Worker-role accounts will fail authz here. | P2 тАФ feature, not bug | Out of scope for v1. Document. |
| R7 | **Spec includes `valid_from`/`valid_to` as `timestamptz` in plan #5 but model serializes as DATE.** See Finding 5. | P0 тАФ schema drift | Verify on staging before writing the migration. |

## Recommendations for the planner

1. **Add a fifth migration** `20260604000050_add_booking_reminder_manual_enum.sql` that runs the same `DO $$ ... ALTER TYPE ... ADD VALUE IF NOT EXISTS` pattern as 20260602130000. Sequence: `000050 тЖТ 000000 (backfill) тЖТ 000100 (reminder RPC) тЖТ 000200 (redeem RPC)`.
2. **Spell out the DTO тЖТ route extras mapping** in the PLAN.md task for the Payment Settings card. Use Finding 1's table verbatim тАФ engineer should not have to grep for `userId` vs. `shopOwnerId`.
3. **Disable the Payment Settings card while `shopDetailsAsync.isLoading || shopDetailsAsync.hasError`.** Otherwise tapping during the first ~200ms after tab open will push with empty strings.
4. **In `schedule_manual_booking_reminder`:** add `delivery_channel` to the INSERT (default `'push'` makes it implicit but be explicit), include `booking_id` in the metadata `rest` payload for deep-linking, and add the past-time guard from R2.
5. **Rewrite `sendBulkReminders` as a single server-side RPC** (`schedule_bulk_manual_booking_reminders(p_shop_id, p_date)`) that does `INSERT INTO scheduled_notifications SELECT ... FROM bookings WHERE ...` in one statement. The spec already hints at this; make it explicit in PLAN.md.
6. **`redeem_promotion`:** switch to UNIQUE-constraint-based idempotency per R4. Cleaner and race-free.
7. **`shop_settings` backfill:** keep minimal (PK shop_id, reminder_settings JSONB, updated_at). Don't over-engineer.
8. **`promotions` backfill:** **verify column types against staging** before writing. The model says DATE; production might be TIMESTAMPTZ. Use whichever is currently live тАФ the goal is parity, not correctness.
9. **`PromotionException` hierarchy:** mirror `WalletException` exactly ([wallet_exceptions.dart:9-27](../../../lib/wallet/data/exceptions/wallet_exceptions.dart#L9)) тАФ `message`, `code`, `userMessage` with the same defaults. Subtypes: `DuplicateCodeException`, `PromotionNotFoundException`, `PromotionLimitReachedException`, `InvalidDiscountAmountException`.
10. **KpiCard:** add an `bool enabled = true` parameter; when false, wrap content in `Opacity(0.5, ...)` and still allow `onTap` to fire (for the SnackBar). Do NOT use `IgnorePointer`.
11. **Document explicitly in PR description:** manual reminders gated only on global `push_enabled`, not per-category `booking_reminders_enabled`. File a P2 follow-up for the worker change.
